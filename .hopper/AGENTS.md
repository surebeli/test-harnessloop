actually have installed and authenticated. Remove rows you do not use.

下方「Dispatch 政策：effort 预制 + model 选择规则」小节）；`--init-tasks --force`

The plugin resolves a task's vendor from the **task-vendor-preference** table
below (or a per-row `Vendor` column in queue.md). `host != vendor` is enforced:
never route a task to the same CLI that is dispatching it.

## Dispatch 政策：effort 预制 + model 选择规则（用户采纳 2026-07-17）

三层结构，各层职责不同，不可混同：

1. **Effort 按 task-type 预制。**
   - 理由：档位枚举（low/medium/high/xhigh 等）是稳定值域，不随时间腐烂；
     评审强度是轮次证据可信度的受控变量，必须固定而非随意波动；固定 effort
     也是成本治理手段之一；grok **无 xhigh 档**，因此 per-vendor 的档位上限
     （grok 封顶 high）必须显性编码在政策里，不能默认所有 vendor 同档。
2. **Model 预制的是「选择规则」而非具体模型名字。**
   - 理由：24 小时内已有两次模型名腐烂的实证——T-001 任务里写死的默认模型名
     在当天即返回 400；若把 `gpt-5.5` 这类名字直接固化进政策，今天就会过时。
   - 规则本身：派发时从 `--check-model` 输出的 **verified** 档清单中，取该
     vendor 当前最新代模型；把**选出的具体名字**钉进实际派发命令行（而不是
     把某个名字钉进本文档）。
3. **实例级可审计性由 hopper 自动保障**，不需要在本文档手工记录每次选中的
   模型名——`output.md` frontmatter 的 `model` 字段 + COST-LOG 会留痕每次
   实际派发用的具体模型与 effort。
4. **偏离政策必须留痕**：任何一次派发若偏离本节规则（例如临时升/降 effort、
   选用了非最新的 verified 模型），必须在 queue 行的 Brief 或对应的
   Activity log 里记一句原因；不留原因的偏离视为误操作。
5. **列语法说明（表格单元格机器可解析语法，2026-07-17 由散文迁移至此）。**
   - 两种记法二选一：**单 token**（如 `medium`）表示对所有 vendor 统一生效；
     **per-vendor 表**（如 `codex:xhigh, grok:high`）表示按 vendor 分别取值，
     逗号分隔、`vendor:值` 结构，未列出的 vendor 视为该列未定义。
   - **`verified-latest` 哨兵**：不钉死具体模型名，派发时动态解析为该
     vendor 当前 `--check-model` 输出中 **verified** 档的 knownGood[0]（最新
     一代 verified 模型）；解析出的具体名字钉进实际派发命令行、并由
     `output.md` frontmatter 的 `model` 字段留痕真名，而不是钉进本文档。
     当前已知解析结果：codex → `gpt-5.6-sol`，grok → `grok-build`（会腐烂，
     以 `--check-model` 实时输出为准，见第 2 条）。
   - **括号开头 `(...)` 记法**（如 `(bind per project)`）表示该任务类型的
     Effort/Model 政策**尚未绑定**——机器应解析为 **unbound**（等项目启用该
     任务类型时再定），而非 unparseable/报错。当前适用：`spec-write`、
     `code-impl`、`sidecar-polish`、`spec-blindspot-hunt`。
   - **`code-review-adversarial` / `code-review-acceptance` 的 effort 说明**：
     `codex:xhigh, grok:high` 中 grok 侧的 `high` 是其**档位上限**（grok
     **无 xhigh 档**），不是随意选的低档；抽中 codex 用 xhigh，抽中 grok
     用 high。
   - **`prd-research` / `market-research` 的 effort 说明**：取 `medium` 是
     因为这两类是**广度型 web-search 任务**；若某次确需深挖，应显式将该次
     派发升为 `high`，并在 queue.md 该任务行记录升档原因（偏离留痕规则见
     第 4 条）。Model 侧 `verified-latest` 沿用同一 vendor 的主力/备选关系
     （grok 主力／codex 备选，见上表 Why 列）。

## Codex 评审强制核对（用户决策 2026-07-17）

已知未修问题（见 `hopper-plugin/ISSUE-codex-review-hijack.md`）：codex 沙箱
**不可靠地降级为只读**（dispatch 传 `read-only`，codex banner 却显示
`workspace-write`）；且存在**跨仓 review 被全局 skill 劫持**的可靠复现记录
（codex 启动自动加载 `using-superpowers`/gstack 等全局 skill，无视 brief 的
范围锁定，审查了错误的仓/目录，产物落到 skill 约定路径而非 brief 指定路径）。

**每次 codex 评审任务完成后，派发方必须核对以下三项，不得跳过：**

1. **实际审查对象**是否确为 brief 指定的目标（正确的仓/commit/路径），而非
   codex 自行推断或被全局 skill 劫持后审查的其他内容。
2. **产物落点**是否落在 brief/dispatch 指定的路径（如
   `.hopper/handoffs/<task>-output.md`），而非 codex 自身 skill 约定的其他
   路径（如某种 `.triage/*.md`）。
3. **不得仅凭 `adapter_status: success` / exit 0 / codex 自述完成即采信**——
   以上两项未核对通过前，视为审查未完成，需重新派发或改人工/主会话直接执行。

## Reassignment

Change a default by editing the table above, or override per task with the
optional `Vendor` column in queue.md. Routing is a static lookup — no
round-robin, no retry-aware rotation. 对 `code-review-adversarial` /
`code-review-acceptance`，"随机"发生在主会话决定并写入 queue 行 `Vendor` 列
的那一步，而不是 hopper 的路由逻辑本身。

## Dispatch rules (capability contract)

Which `--model` / `--reasoning` / `--sandbox` / `--timeout` each vendor honors —
plus its permission + working-dir mapping and timeout model — lives in
`.hopper/DISPATCH.md`. That file is **GENERATED from the hopper adapters; do NOT
hand-edit it, and do NOT hand-copy vendor invocation strings elsewhere** (they rot).
Regenerate after upgrading hopper: `hopper-dispatch --rules > .hopper/DISPATCH.md`.
Live single-vendor check: `hopper-dispatch --capabilities <vendor>`.
